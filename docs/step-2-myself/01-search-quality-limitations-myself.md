# 01. 현재 프로젝트에서 키워드 검색 한계 확인하기

이 문서는 강의자료 `docs/step-2/01.search-quality-limitations.md`를 현재 프로젝트 기준으로 보강한 내용이다.

원본 문서는 키워드 검색의 한계를 개념적으로 설명한다. 구현 전에 이 한계를 실제 데이터와 현재 검색 코드에서 확인해 두면, 이후 벡터 검색과 하이브리드 검색이 정말 개선인지 판단할 수 있다.

## 현재 기준선

현재 검색 흐름은 아래와 같다.

```text
BookSearchController
  -> BookSearchService
      -> BookRepository.search(...)
          -> BookRepositoryImpl QueryDSL LIKE 검색
```

검색 대상 필드는 현재 `BookRepositoryImpl` 기준으로 아래와 같다.

```text
title
authorName
publisherName
subtitle
category
bookContent
isbn
```

`keyword`는 여러 필드에 `containsIgnoreCase`로 적용되고, `isbn`은 정확 일치로 적용된다. 이 구조는 구현이 단순하지만 사용자가 정확한 단어를 입력하지 않으면 결과가 쉽게 빠진다.

## 왜 기준선을 먼저 잡아야 하는가

벡터 검색을 붙인 뒤 “좋아진 것 같다”로 끝내면 어떤 문제가 해결됐는지 알기 어렵다. Step 2 시작 전에 키워드 검색이 실패하는 예시를 기록해 두는 편이 좋다.

기준선은 아래 정도면 충분하다.

| 구분 | 예시 | 키워드 검색 예상 문제 |
| --- | --- | --- |
| 동의어 | `컴퓨터 구조` vs `Computer Architecture` | 글자가 다르면 못 찾을 수 있음 |
| 표현 차이 | `입문자를 위한 자바` vs `자바 기초` | 의미는 비슷하지만 순위가 낮을 수 있음 |
| 자연어 질문 | `처음 프로그래밍 배우는 사람이 볼 책` | 특정 키워드가 없으면 결과가 약함 |
| 고유명사 | `한강`, `토비`, `자바의 정석` | 키워드 검색이 오히려 강함 |
| ISBN | `978...` | 키워드보다 정확 일치가 맞음 |

## 간단한 검색 품질 기록표

처음부터 정밀한 평가 시스템을 만들 필요는 없다. 수동으로라도 아래 표를 남기면 이후 변경의 효과를 비교하기 쉽다.

| 검색어 | 의도 | KEYWORD 상위 결과 | 문제 | VECTOR/HYBRID에서 기대하는 것 |
| --- | --- | --- | --- | --- |
| `자바 입문` | Java 초보 도서 | 제목에 자바가 있는 책 | 입문/기초 의미 구분 약함 | 입문서가 상위에 와야 함 |
| `객체지향 쉽게 설명한 책` | OOP 입문서 | 결과 적거나 없음 | 자연어 처리 불가 | OOP/Java 기초 도서 검색 |
| `컴퓨터가 내부에서 동작하는 원리` | 컴퓨터 구조 | 키워드 없으면 약함 | 의미 검색 필요 | 컴퓨터 구조/운영체제 도서 |
| `978...` | 특정 도서 | 정확히 1건 | 문제 없음 | 키워드 검색 우선 |

## 현재 단계에서 추가하면 좋은 테스트

Repository 테스트에는 검색 조건 자체를 검증하는 케이스를 먼저 둔다.

```text
1. keyword가 title에 포함되면 조회된다.
2. keyword가 authorName에 포함되면 조회된다.
3. isbn은 정확 일치로 조회된다.
4. keyword와 isbn을 같이 주면 AND 조건으로 좁혀진다.
5. keyword가 없으면 전체 목록이 페이징된다.
```

이 테스트는 벡터 검색 품질을 검증하지 않는다. Step 2 이후 검색 타입을 늘릴 때 기존 키워드 검색이 깨지지 않았는지 확인하는 안전장치다.

## 벡터 검색이 해결하지 못하는 것

원본 문서는 벡터 검색의 장점을 강조하지만, 실제 구현에서는 한계도 같이 봐야 한다.

| 상황 | 벡터 검색 한계 | 대응 |
| --- | --- | --- |
| ISBN 검색 | 의미 검색 대상이 아님 | `isbn`이 있으면 정확 일치 우선 |
| 저자명/출판사명 | 고유명사는 임베딩이 약할 수 있음 | 키워드 검색과 병합 |
| 최신 신조어/오타 | 모델이 모르면 품질이 낮음 | keyword fallback 유지 |
| 데이터 전처리 불량 | HTML, 특수문자가 의미를 흐림 | 임베딩 전 전처리 |
| 임베딩 누락 | 벡터 검색에서 제외됨 | 누락 데이터 재처리 배치 |

따라서 Step 2의 목표는 키워드 검색을 버리는 것이 아니라, 키워드 검색이 약한 자연어/의미 검색을 보완하는 것이다.

## 구현 기준

현재 프로젝트에서는 아래 기준으로 진행하는 것이 가장 덜 헷갈린다.

```text
KEYWORD:
  기존 QueryDSL LIKE 검색 유지

VECTOR:
  검색어를 embedding으로 변환한 뒤 embedding 컬럼과 거리 계산

HYBRID:
  KEYWORD 결과와 VECTOR 결과를 각각 가져와 RRF로 병합

RAG:
  Step 3 이후에 붙임
```

Step 2에서 `RAG`를 먼저 기본값으로 두면 문제가 생겼을 때 원인이 검색인지 LLM인지 구분하기 어렵다. 현재 `BookSearchRequest` 기본값처럼 `KEYWORD`를 기본 검색으로 두고, 사용자가 선택했을 때 `VECTOR`와 `HYBRID`를 타게 하는 구성이 좋다.
